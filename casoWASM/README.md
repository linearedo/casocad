# casoWASM

Rust port of casoCAD targeting **native desktop and the web browser** from a
single codebase (wgpu). The existing Python casoCAD is never modified; this
workspace lives beside it and proves parity against it.

Principles:

- `#![forbid(unsafe_code)]` in every crate (enforced via workspace lints).
- Minimal dependencies: the kernel has **zero**; later crates add only what is
  irreplaceable (`wgpu`, `egui`).
- f64 for all kernel/analysis math; f32 only at GPU upload boundaries.
- SDF terminology: boolean operations are *SDF operators* (never "CSG");
  "meshing" is reserved for FEA/CFD — the viewport makes *surfaces*.

## Crates

- `kernel/` — the exact signed-distance-field geometry kernel (port of
  `core/`): primitives (1D/2D/3D), SDF operators, exact transforms and
  generators, the exactness role system ("safe geometry compiler", spec
  `design_docs/exact_signed_distance_field_cfd_migration_v2.md`), Model
  compile (grammar + preconditions + Domain disjointness).

Planned next (see the port plan): `surfaces/` (display-surface builders:
exact boolean clipping + dual contouring), `render/` (wgpu, WGSL), `app/`
(egui + egui_dock), `web/` + `native/` entry points, meshing workspace.

## Parity against Python casoCAD

Golden files are generated by the Python kernel and replayed by Rust tests:

```bash
# from the casoCAD repo root (regenerates kernel/tests/goldens/)
.venv/bin/python casoWASM/tools/export_goldens.py

cd casoWASM
cargo test          # includes the 52-fixture / ~11.5k-sample parity suite
cargo clippy --all-targets
cargo build --target wasm32-unknown-unknown
```

Every fixture in `kernel/tests/parity.rs` must match the Python `to_numpy()`
values to a 1e-12 relative tolerance. When adding a node kind, extend both
`tools/export_goldens.py` and `build_fixture` in the test, then regenerate.
