# FFI layer for multi-language bindings (Python, Julia, C++)

*Distilled from a design dialogue between edo and Claude (2026-07-11).
Status: agreed direction, nothing implemented yet.*

## 1. Context and settled ground

- The Meshing workspace scripting question is **settled**: Rhai in-app is
  good enough (runs identically on web and native); power meshing happens
  **outside the app** through the file contract — `scene.json` in →
  `.arrow` (Arrow IPC mesh artifact) out. No embedded Python, no Pyodide.
- The future need: a `casocad` package per language (Python first, Julia,
  maybe C++) that loads a scene.json and exposes the Fluid/Solid Domain
  signed-distance functions plus boundary regions — the analysis API, for
  writing external meshers/tools.
- Key fact about the kernel: the SDF of a domain is **data, not code** — a
  Rust `Shape` enum tree (`casoWASM/kernel/src/sdf/node.rs`) evaluated by
  one fixed interpreter (`Node::eval_point`): operators are `min`/`max`/
  `max(a,−b)`, transforms inverse-map the point, primitives bottom out in
  closed-form distance formulas (parity-tested 1e-12 vs Python). Any
  binding ultimately calls this tree walk.

## 2. Decision: one C ABI, N thin wrappers

Don't write N language bindings against Rust; write **one C ABI crate** and
let every language bind C — the lingua franca all targets consume natively.

```
caso-kernel (pure Rust, forbid(unsafe), untouched)
      ↑
caso-ffi  (new crate: cdylib, C ABI — the ONLY place unsafe exists)
      ↑ cbindgen → casocad.h
      ├── C++    : includes casocad.h (+ optional small RAII header)
      ├── Julia  : @ccall against libcasocad — first-class C FFI, no glue
      └── Python : cffi over the same .so (maturin bindings="cffi");
                   a PyO3 crate is optional ergonomic sugar later
```

Adding language N+1 costs a thin idiomatic wrapper in *that* language,
never new Rust.

## 3. Scope discipline (what the ABI exposes)

Only the **analysis contract**, mirroring `MeshableDomain`
(`casoWASM/kernel/src/meshing.rs`). Scene *editing* stays out — it's the
app's job and the churny part of the API. Roughly a dozen functions:

```c
CasoScene*   caso_scene_load_json(const uint8_t* bytes, size_t len);
CasoDomains* caso_domains_create(const CasoScene*);   // runs compile_model gate
CasoDomain*  caso_domain_get(const CasoDomains*, const char* name_or_kind);
void         caso_domain_bounds(const CasoDomain*, double out[6]);
int32_t      caso_domain_sdf(const CasoDomain*, const double* xyz, size_t n, double* out);
int32_t      caso_region_contains(const CasoRegion*, const double* xyz, size_t n, uint8_t* out);
int32_t      caso_region_owner_sdf(const CasoRegion*, const double* xyz, size_t n, double* out);
const char*  caso_last_error(void);
uint32_t     caso_abi_version(void);
/* + one caso_*_free per handle type */
```

## 4. ABI design rules (each makes wrappers trivial)

- **Opaque handles only** — no structs cross the boundary; Rust internals
  stay free to change.
- **Batch, caller-allocated, flat f64 buffers** — one FFI call per million
  points; zero-copy from numpy / Julia arrays / `std::span`. (The in-app
  Rhai shim is per-point interpreted dispatch — the batch ABI is also the
  performance story.)
- **Error codes + `caso_last_error()`** — translates to native exceptions
  in every language.
- **`caso_abi_version()`** checked by every wrapper at load time.
- `unsafe` confined to `caso-ffi` (its own `[lints]`, outside the
  workspace `forbid(unsafe_code)`) — the FFI boundary is the one visible,
  contained exception.
- Crate name `caso-ffi` but `lib.name = "casocad"` → the artifact is
  `libcasocad.so/.dylib/.dll`, matching the header and every wrapper's
  dlopen string forever.

## 5. Repository layout (agreed)

```
casoCAD/
├── casoWASM/
│   └── ffi/                      # caso-ffi crate (cdylib + staticlib)
│       ├── src/lib.rs            # the ~dozen extern "C" functions
│       ├── cbindgen.toml
│       ├── include/casocad.h     # generated by cbindgen, COMMITTED
│       └── tests/smoke.c         # compiled + run in CI against the header
├── bindings/                     # one folder per language, no Rust inside
│   ├── python/                   # → PyPI package "casocad"
│   │   ├── pyproject.toml        # maturin, bindings = "cffi"
│   │   └── casocad/__init__.py   # idiomatic API, numpy in/out
│   ├── julia/                    # → CasoCAD.jl (Project.toml, @ccall)
│   └── cpp/                      # casocad.hpp RAII wrapper + CMakeLists
└── .github/workflows/bindings.yml  # wheel matrix + C smoke + julia test
```

- `ffi/` lives in the casoWASM workspace: versions/builds/tests with the
  kernel it wraps.
- Wrappers live outside the workspace, each shaped for its ecosystem's
  tooling (pip/maturin, Julia registrator subdir packages, CMake).
- Dependency arrow strictly one-way: wrappers know the header + ABI
  version, nothing else. Monorepo now; a binding can be split to its own
  repo later painlessly (Julia community prefers that eventually).

## 6. Python user experience (the target)

`pip install casocad` — no Rust toolchain, no compiler. The wheel bundles
the compiled `libcasocad` (like numpy ships BLAS); CI builds wheels per
platform (manylinux x86_64/aarch64, macOS x86/arm64, Windows) and pip
picks the right one. `maturin generate-ci` writes the workflow.

```python
import casocad, numpy as np
scene  = casocad.load_scene("scene.json")
fluid  = scene.domains["fluid"]
points = np.random.rand(1_000_000, 3)      # (N, 3) float64
d      = fluid.sdf(points)                 # ONE FFI call, zero-copy, ndarray out
mask   = fluid.regions["inlet"].contains(points)
```

At import: locate bundled lib via importlib.resources → cffi dlopen →
check `caso_abi_version()` → wrap handles in idiomatic classes; errors
become Python exceptions; GC frees handles.

## 7. Publishing (checked 2026-07-11)

- **PyPI is completely free** (accounts, hosting, bandwidth). 2FA is
  mandatory. Practice on TestPyPI first; releases are immutable.
- **The name `casocad` is available on PyPI** (verified 404 on
  pypi.org/pypi/casocad/json, 2026-07-11). Publishing a minimal 0.0.1
  early legitimately reserves it.
- Use **Trusted Publishing** (GitHub OIDC, no stored tokens).
- Julia General registry: free (a PR). C++: GitHub release artifacts.
  Nothing in the plan has a distribution cost beyond CI minutes.

## 8. Rejected alternatives (don't re-propose)

- **Embedded Python / Pyodide in the app**: 20–30 MB wasm download, no
  usable gmsh build, poor perf for lattice loops, dual scripting backends
  forever — and the workflow doesn't need it (solvers are native anyway;
  the web app's meshing role is authoring/preview).
- **A fresh pure-Python reimplementation of the SDF math**: a third copy
  of the formulas — exactly the drift the golden-file harness exists to
  prevent. (The frozen Python `core/` remains usable as a reference/
  stopgap library, but the C ABI path supersedes it.)
- **Per-language native bindings as the primary strategy** (PyO3 + jlrs +
  cxx…): N× maintenance; PyO3 stays only as optional sugar over the same
  contract.
- **WASM component model as the universal binary**: portable and sandboxed
  but slower, heavier, and every host integration is younger than C FFI —
  wrong trade for a numeric kernel called in tight loops.
